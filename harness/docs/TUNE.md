# TUNE — финальный тюн: сопоставить ситуацию ↔ несоответствие, оттрассировать ложные фиры

Харнес после bootstrap работает, но лексика сырая. Этот цикл превращает fire-log из
«потока строк» в датасет с генеалогией и находит два вида ошибок:

- **arrival-miss** (дыра лексикона): ядро РЕАЛЬНО легло в ответ, но промпт его не зажёг →
  фраза промпта = кандидат нового триггера.
- **echo** (ложный фир): спан зажёг ядро, но это слово-тема (разговор ПРО термин), не акт →
  спан = кандидат анти-триггера.

## Цикл (порядок жёсткий)

```sh
G=~/.claude/glados

# 0. Пожить с харнесом. Fire-log копится сам (ida-attest пишет ход, ida-land доливает
#    acted_on+witness по ответу). Меньше ~50 содержательных ходов — тюнить рано.

# 1. Ретро-заполнить старые строки (acted_on=null до включения ida-land):
python3 tools/backfill_acted.py            # dry-run отчёт
python3 tools/backfill_acted.py --apply

# 2. Лог → датасет:
python3 tools/compile_cases.py             # → $G/case-book.jsonl
#    ⚠ перезаписывает case-book: после recompile повторить merge anno (шаг 4).

# 3. LLM-судья (в Claude Code): «прогони workflow/annotate.workflow.js на $G/case-book.jsonl»
#    → судья даёт verdict genuine/echo/partial per (кейс, ядро) → движок пишет annos.json.

# 4. Мерж вердиктов:
python3 tools/annotate_merge.py annos.json --src judge-$(date +%F)

# 5. Кандидаты тюна (оба = отчёт + jsonl, НИЧЕГО не правят сами):
python3 tools/casebook_harvest.py          # arrival-miss × genuine → harvest-candidates.jsonl
python3 tools/casebook_sidecases.py        # echo-спаны → anti-trigger-candidates.jsonl
                                           # + exemplars.jsonl (few-shot банк судьи)

# 5.5 ГЕЙТКИПЕР — перед ЛЮБОЙ правкой матрицы (обязательный шаг):
python3 tools/rebalance_check.py          # пропорции vs снапшот, echo ПО ЯДРАМ,
                                          # утечка диспозиций в матрицу, гигиена
#    Правило: нельзя тюнить число, не разложенное по классам ядер.
#    Echo высокий на СИТУАЦИЯ-ядре = проблема; на диспозиции = by design, НЕ трогать.
#    Живой прецедент: «почини 51%» чуть не убил слой — CASES.md §5.
#    После осознанной правки: rebalance_check.py --save (новый бейзлайн).

# 6. ЧЕЛОВЕК смотрит кандидатов и правит core-triggers.txt руками:
#    harvest-кандидат → добавить слово/фразу в regex ядра;
#    анти-триггер     → сузить regex (negative lookahead / убрать слово).
#    После правки:
python3 build_embeddings.py --triggers $G/core-triggers.txt --out $G/core_p2p.json
```

Дальше цикл повторяется по мере жизни: шаги 2–6 раз в неделю-две.

## Почему судья обязателен (число, не вкус)

Оракул acted_on = **reply-lex**: тот же лексикон сканирует ответ модели. Он смещён самим
собой — на референс-вульте (651 вердикт) **echo-rate = 51%**: половина «фиров» — слово-тема,
не акт. Доминантный ложный мод: мета-разговор про сам харнес зажигает его же лексику.
Дешёвый механический флаг ЭТО не отделяет (замерено: meta-flag echo 52% vs 51% — не
разделяет). Значит:

- любой авто-тюн по сырому acted_on наследует ~50% шума → **фильтр по anno.verdicts
  обязателен** (шаги 5 берут только genuine / только echo);
- вердикт = семантическая работа → LLM-судья, не regex.

## Три правила безопасности цикла

1. **Рекурсию не замыкать.** Лексикон породил acted_on → acted_on предлагает лексикон.
   Авто-врезка = self-fulfilling шум. Врезку в core-triggers.txt всегда триггерит человек.
2. **Источник едет с меткой.** `acted_src` (reply-lex / reply-lex-backfill / human-label),
   `anno_src` (какой судья, когда), `voice` (human/paste). Пост-хок анализ обязан различать;
   human-label никогда не перезаписывается оракулом.
3. **Paste — не диалект.** Длинный вставленный блок (`voice=paste`) — чужой текст;
   харвест лексикона его не берёт (casebook_harvest уже фильтрует).

## Третья ось (act/read) — тонкая настройка после первых недель

`dir` в fire-log = направление промпта (act «сделай/положи» vs read «покажи/где») из
грамматики, не вокабуляра. Если по casebook видно, что ядро X ложно фирит на read-формах
(его триггеры = act-глаголы), впиши его в `$G/axis.json`:

```json
{"act_native": ["5", "8"]}
```

— и ida-attest замьютит read-фолс-фиры этого ядра. По умолчанию файл пуст: сначала
докажи по датасету, что мьют нужен, потом включай.

## Два класса ядер — куда какое (важно ДО тюна лексики)

- **Ситуация-ядра** (pull, зажиг = слово): живут в лекс-матрице, их триггеры этот цикл
  и тюнит. Харвест их лексику РАСШИРЯЕТ — можно.
- **Диспозиция-грани** (push, применимы ≈ каждый ход): лексикой их диспетчить НЕЛЬЗЯ —
  similarity зашумит (они везде), regex недоберёт. Их место — **пол**: пометь файл ядра
  `class: disposition` во frontmatter (или впиши путь в `~/.claude/glados/floor.txt`) —
  `ida-floor` инжектит ТЕЛО при старте сессии. Триггеры диспозиций не харвестить.
- Симптом неверного класса: ядро фирит почти каждый содержательный ход → это диспозиция,
  убери его строку из core-triggers.txt и переведи в пол.

## Что считается успехом

Не «больше фиров» (это recall-by-construction, не победа). Успех =
- arrival-miss убывает на новых ходах (лексикон догнал диалект);
- echo-rate убывает между раундами разметки (анти-триггеры сработали);
- surfaced-ядра, подтверждённые genuine, растут в доле.

Все три числа считаются из case-book — датасет и есть измерительный прибор.
