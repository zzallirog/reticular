# ATLAS-LAYOUT — раскладка памяти под ТВОЮ топологию проектов

Харнес (ядра+лексика) ловит момент. Но чтобы агенту было ЧТО вспоминать и КУДА смотреть,
нужна раскладка рабочей памяти. Основа — **agent-atlas**:

> https://github.com/zzallirog/agent-atlas
> Модульная раскладка working-memory для агента, работающего поверх многих реп.
> Plain Markdown, provider-agnostic. Склонируй и прочти README + docs/ — это канон.

Опционально, когда нот станет много: https://github.com/zzallirog/memory-atlas —
граф всего вульта одним self-contained HTML (видно дыры и дубли раскладки).

Здесь — только то, что atlas не говорит: как раскладка СЦЕПЛЯЕТСЯ с лекс-матрицей харнеса.

## Принцип сцепки: имя обязано лексически выражать направление

Лекс-матрица — это regex по словам. Она работает ровно настолько, насколько твои файлы,
проекты и feedback-ноты **называются словами, которыми ты о них говоришь**. Правила:

1. **MEMORY.md = индекс, не контент.** Одна строка на ноту: `- [имя](файл.md) — крючок`.
   Крючок — 5-10 слов ТВОЕГО диалекта (те же слова, что ты скажешь в промпте). Индекс
   грузится на старте; тела нот — on-demand.
2. **Одна нота = один факт/проект.** Frontmatter: `name` (kebab-slug), `description`
   (по ней решается релевантность при recall), `type` (user/feedback/project/reference).
3. **Feedbacks репрезентуют проекты.** Каждая feedback-нота отвечает: на КАКОМ проекте
   обожглись, ПОЧЕМУ, КАК применять (`**Why:**` / `**How to apply:**`). Feedback без
   привязки к проекту — мёртвый груз: лексика проекта его никогда не поднимет.
4. **Направление — в самом названии.** `feedback_verify_before_push` поднимается на слове
   «push»; `feedback_note_17` — никогда. То же для проектов: `proj_<имя-как-ты-его-зовёшь>`.
5. **Ссылки `[[имя]]` между нотами** — граф. Битая ссылка = «нота, которую стоит написать»,
   не ошибка.

## Слои (снизу вверх)

| Слой | Файл(ы) | Когда в контексте |
|---|---|---|
| Раскладка репо | atlas-навигаторы в каждом репо | при входе в репо |
| Индекс памяти | MEMORY.md | всегда (старт сессии) |
| Ноты | memory/*.md | on-demand (recall/ссылка) |
| Пол (диспозиции) | cores с class: disposition + floor.txt | всегда (SessionStart, ida-floor) |
| Ядра (ситуации) | ~/.claude/overlays/cores/*.md | по фиру лексики (ida-attest) |
| Лекс-матрица | ~/.claude/glados/core-triggers.txt | каждый промпт (хук) |
| Fire-log/casebook | ~/.claude/glados/*.jsonl | никогда (это измерение, не контекст) |

Ядра указывают «в такой ситуации бывают такие провалы — перечитай»; ноты несут факты;
атлас несёт раскладку реп. Три разных вопроса — три слоя, не один файл.

## Порядок наведения (для существующей памяти)

1. Склонируй agent-atlas, прочти, разложи навигаторы по своим репо.
2. Пройди свои feedback/заметки: каждой — name/description/type + привязка к проекту.
   Что ни к чему не привязано — в архив.
3. Перепиши крючки MEMORY.md СВОИМИ словами (как ты реально это называешь в промптах).
4. Только потом bootstrap харнеса: harvest берёт лексику из твоих логов, и чем чище
   раскладка — тем чище эмерж ядер.
